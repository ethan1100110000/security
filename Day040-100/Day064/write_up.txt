# Day64 Heap - UAF 1

## Goal

`free()` 이후 남은 dangling pointer가 새로 할당된 객체를 조작할 수 있음을 확인했다.

## Code Flow

```c
User *a = malloc(sizeof(User));
User *alias = a;

free(a);
a = NULL;

User *b = malloc(sizeof(User));
strcpy(alias->name, "HACKED_BY_ALIAS");

a = NULL을 해도 alias에는 old heap 주소가 남아 있다.

같은 size class로 b를 할당하면 allocator가 같은 chunk를 재사용할 수 있고, 이때 alias == b가 된다.

Result
alias == b
alias write → b->name overwrite

즉 stale alias pointer를 이용한 UAF write를 재현했다.

Root Cause

free() 이후에도 같은 heap chunk를 가리키는 alias pointer가 남아 있고, 그 포인터를 다시 사용하는 코드 흐름이 문제다.

Lifetime 관점

free(a) 순간 기존 User 객체의 lifetime은 끝난다.
주소값이 남아 있어도 그 객체에 접근하는 것은 UAF이며 undefined behavior다.

Failure Case

b를 다른 size class로 할당하면 같은 chunk가 재사용되지 않아 alias == b가 안 될 수 있다.

User *b = malloc(0x100);

하지만 dangling pointer가 남아 있으므로 UAF bug 자체가 사라진 것은 아니다.

Summary
free()는 포인터 값을 지우지 않는다.
a = NULL은 alias pointer를 해결하지 못한다.
같은 size class malloc은 free된 chunk를 재사용할 수 있다.
stale alias pointer로 새 객체를 덮으면 UAF write가 된다.
