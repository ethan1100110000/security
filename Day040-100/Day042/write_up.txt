checksec 으로 보호기법을 확인했고 ropgadget에서 ret 가젯이 있는것을 확인
objdump 에서 main vuln win 함수가 있는걸 확인

##목표
카나리값 leak bof로 win 함수 호출

find.py프로그램으로 카나리 후보인덱스를 먼저 추렸음 이번 실습에서 후보는 16 23 27 47
후보 확정을 위해 gdb에서 printf에 브레이크를 걸고 x/gx rbp-0x8 로 먼저 이번 실행의 카나리값을 확인
이후 후보들을 입력에 넣어서 출력 확인 미리 본 카나리값과 같은지 비교 23 27 47 의 값들이 카나리값과 같은걸 확인했다
안정적인 실행을 위해 카나리 인덱스는 가장 작은 23으로 확정

카나리 offset 확인 disas vuln에서 두번째 read의 rbp-000 를 확인 왜? 저 값이 buf의 시작 즉 buf에서 saved rbp 까지의 거리
000이 온전히 offset인건 아니다 -0x8 을 해야 패딩의 크기가 된다 8바이트의 카나리값이 들어가야 하기 때문에
이번 실습해서는 rbp - 0x50인것을 확인 0x50은 10진수로 80 여기에 0x8을 빼면 72 즉 패딩의 크기는 72

##payload 구조
1차는 fsb 첫 입력에서 %1$p를 넣어봤을때 주소의 형태로 출력이 나오는것을 확인 fsb 취약점 있음
2차 입력에서 a를 200개 정도 넣어봤을때 stack smashing detected 출력 즉 카나리 값이 덮힘 이 말은 bof 발생 취약점 확인

1차에선 f"%{canary_idx}$p"encode() 로 카나리값을 leak한다 16진수 정수형으로 파싱
pie는 꺼져있기 때문에 ret, win 의 주소를 미리 구해놓고
2차 패딩(72바이트) + 카나리 + saved rbp를 덮을 8바이트 더미 + ret(필요시 추가 win 함수 진입시에 rsp 정렬이 깨지는지) + win

##실패
WIN 출력은 되지만 쉘에서 명령 입력 실패
ret이 없는 상태에서 실패 카나리값을 먼저 확인 파싱한 카나리값 출력을 프로그램에 적어 놓았으니 gdb에서 2차 페이로드가 입력된 바로 다음 즉 read함수 바로 다음 주소에 브레이크를 걸고 rbp-0x8 확인 카나리값과 같다 카나리값은 문제 없다고 판단
rbp값은 더미값 rbp + 0x8은 win 주소 값이 잘 들어가있는데 실패 사실 WIN 출력이 되었으니 win 함수 진입까지는 성공이기때문에
위에서 본 것은 크게 의미가 없다는 생각이 들지만 혹시 모르니 일단 봤음 rsp % 16값이 0x0인것을 확인 즉 stack aligment
 페이로드에 ret 추가 ret 추가 후 스택구조는 rbp + 0x8 = ret rbp + 0x10 = win
win 출력 쉘에서 명령입력 성공
