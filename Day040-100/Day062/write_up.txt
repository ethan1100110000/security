# Day62 Heap Metadata Write-up

Day62에서는 glibc heap chunk의 metadata 구조를 gdb로 확인했다.

malloc이 반환하는 포인터 `p`는 chunk의 시작 주소가 아니라 user data 시작 주소다.  
metadata는 user pointer 앞에 존재한다.

```text
p - 0x10 : prev_size
p - 0x08 : size | flags
p        : user data

chunk size는 0x10 단위로 정렬되기 때문에 size field의 하위 4비트는 flag로 사용된다.

real_size  = size_field & ~0xf;
flags      = size_field & 0xf;
PREV_INUSE = size_field & 1;

PREV_INUSE는 현재 chunk가 아니라 이전 chunk가 사용 중인지 나타낸다.

PREV_INUSE = 1 → 이전 chunk 사용 중
PREV_INUSE = 0 → 이전 chunk free 상태

prev_size는 항상 의미 있는 값이 아니며, PREV_INUSE=0일 때만 이전 chunk 위치 계산에 사용된다.

실습에서는 malloc(0x30)의 실제 chunk size가 0x40이고, size|flags가 0x41인 것을 확인했다.
또한 malloc(0x80)은 실제 chunk size가 0x90이고, size|flags는 0x91이었다.

free(a) 이후에도 다음 chunk의 PREV_INUSE가 바로 0으로 바뀌지 않았다. 작은 chunk는 tcache에 들어가며 즉시 coalescing되지 않을 수 있기 때문이다.

이후 같은 크기로 다시 malloc(0x30)을 호출했을 때 d == a가 되는 것을 확인했다. 이는 free된 chunk가 tcache에 저장되었다가 같은 size class 요청에서 재사용되었기 때문이다.

따라서 이미 free된 dangling pointer a로 write하면, 같은 chunk를 재사용한 d의 data를 덮을 수 있다. 이것이 UAF write의 기본 구조다.

또한 free된 chunk의 user data 영역에는 tcache next/key 같은 allocator 내부 값이 남을 수 있으므로, dangling pointer로 읽으면 heap/allocator 정보 leak으로 이어질 수 있다.

정리하면, 오늘은 heap chunk의 prev_size, size|flags, PREV_INUSE, tcache 재사용, UAF write/read의 기본 원리를 gdb로 확인했다.
